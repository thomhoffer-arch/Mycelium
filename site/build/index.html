Build a connector

A connector is a small program that reads one source — Revit, ArchiCAD, ERPNext, BCF, email, an in-house tool — and outputs records in the shape Mycelium expects. Other tools can then join your data with theirs.

What you emit

Two objects per record. That's the whole contract.

{
  "identity":  {
    "source": "erpnext",
    "sourceLocalId": "PUR-ORD-2026-00005",
    "projectKey": "horizons",
    "classification": [{ "system": "NL-SfB", "code": "23.22" }],
    "workPackage": "WP-BALCONY-L3"
  },
  "freshness": {
    "source": "erpnext",
    "revisionId": "2026-06-12T10:00:00Z",
    "confidence": "snapshot"
  }
}

Identity is a set of keys, not just one. Anyone reading it joins on whichever key they also have. Model tools join on uniqueId or ifcGuid; finance and field tools join on classification, workPackage, or zone. Freshness is the revision the data belongs to so a consumer can refuse a stale join.

Five steps

  1. Read the spec. It defines identity, freshness, provenance and how joins are supposed to work.
  2. Decide your mapping. Which upstream field is the stable id? Which fields become which identity keys? The spine adapter template walks through this.
  3. Write the connector. Use lib/spine-adapter.mjs — zero dependencies, self-certifying. You supply fetchSource(); it handles the rest. The reference connector and the email example are short enough to read in one sitting.
  4. Check conformance. Run node conformance/validate.mjs on a sample, and work through the conformance checklist.
  5. Register it. Add it to the hub — either suggest the source (no code) or open a PR adding a row to hub/REGISTRY.md when your connector is ready.

If your connector also writes

Most connectors only read. If yours also writes back — flips a clash status, files an RFI, posts a cost item — every change goes through propose → approve → execute, and each step is recorded as an event in an append-only, hash-chained log (lib/provenance.mjs). Actors in the log are pseudonymous refs like human:u_8f3a or agent:llm, never names or emails.

Maturity

  • experimental — works on a fixture; not battle-tested.
  • community — in real use somewhere; not independently verified.
  • verified — passes the conformance kit and has been tested on a real project. Prefer this in production.
