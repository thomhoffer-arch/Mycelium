Why your tools don't talk — and the open fix.

BIM coordinator chasing a clash that finance somehow already paid for. Project manager looking at six dashboards that disagree about the schedule. Foreman holding a tablet whose model is two revisions old. Cost surveyor exporting CSVs because nothing bridges the takeoff and the PO. FM manager handed a model and a folder of PDFs at handover. Owner asking who decided this. If any of those is you — read on.

"Why are my Revit ElementIds different from my colleague's?"

Because ElementId is local to a Revit document. Two copies of the same file have different ElementId spaces. The Revit UniqueId is the stable key across worksharing, and the IFC GlobalId can be derived from it deterministically — no IFC export required. Mycelium ships the derivation as a zero-dependency function so every tool produces the same GUID without having to run the export round-trip.

"Why does my IFC export give different GUIDs than the stored IFC_GUID parameter?"

Because the stored IFC_GUID Revit parameter is only populated after an IFC export round-trip, and is inconsistently present across project files. Mycelium's spec explicitly deprecates the stored param as a cross-tool key. Use the UniqueId, derive the IfcGuid; everything matches.

"Why don't my clash, my model and my purchase order line up?"

Because each tool has a different idea of "the same thing." The clash detector knows a GUID. The ERP knows an NL-SfB classification and a work-package. The field inspector knows a zone. None of them share a primary key. Mycelium's identity record is a set of keys — GUID, classification, work-package, zone — so anyone can join on whatever they have.

"Why am I paying €40k a year for a cloud I don't trust with my drawings?"

Because the vendor cloud is currently the only thing that joins your tools — so you're paying for the join, not the cloud. Mycelium aims to remove that as a paid precondition: the contract is open, the connectors are open, and a tool that speaks the contract can read your data wherever the data already lives. You can still buy a cloud if you want one; you no longer have to. See Trust for the four buyer fears, addressed.

"Who decided this? Who approved it? When? Can I prove it later?"

Every approved change in the network is appended to a tamper-evident provenance ledger (SHA-256 chain, pseudonymous actors, GDPR-friendly). Six months from now an insurer or a court can verify the chain in seconds. No vendor between you and the evidence.

"My PM tool, my ERP and my schedule never agree. Whose number do I trust?"

Because each tool is reading a different snapshot, and there is no shared way to say so. The spec defines a freshness stamp every record carries — which revision, when, live or snapshot. That gives any tool consuming the data the information it needs to compare revisions before merging anything. Mycelium doesn't run your project; it just makes "the same revision" a thing tools can agree on.

"I'm the foreman with a tablet. Why is the model in front of me out of date?"

Because the model lives in one tool, your field tool in another, and there is no shared notion of "what revision is on site right now." Once the data carries a freshness stamp, any field app — open-source, in-house, or commercial — can act on it. Mycelium provides the shared notion; what your foreman sees is decided by the app.

"Handover gave me a model and a hundred PDFs. How am I supposed to maintain this?"

Because design data and operations data were never meant to meet. The spec keeps identity (uniqueId, classification, zone) on the records all the way through — design, build, handover, FM — so the asset register, the warranty PDFs and the maintenance schedule can join back to the element that was actually installed. Whether they do is then a question of which FM tool you pick — but the data layer is no longer the reason it's impossible.

So what is Mycelium, exactly?

A small, open contract — three things: identity, freshness, provenance — that every tool in the chain can speak. Build a connector once; any tool that speaks Mycelium can read your data. No vendor cloud in the middle. Apache-2.0. EU-rooted.

And what it isn't.

Mycelium does not run your project, write your reports, or decide which clash matters. It is the data layer underneath those things — the part that's missing today. The tools that actually do the joining, the triage, the writing — that's an orchestrator layer above the spine. Mycelium makes those tools possible (and interchangeable); it doesn't pretend to be one.

Beyond AEC

Mycelium was built for AEC because that's where the data fragmentation hurts most right now. The contract itself is not AEC-specific: identity, freshness and provenance are properties of any operational dataset that lives in multiple tools.

Adjacent fits we haven't built connectors for yet, but where the same standard would slot in cleanly: warehouses (WMS, asset tracking, equipment maintenance), commercial offices (FM, space management, IoT/BMS), hospitals (asset lifecycle, BMS, compliance), factories (digital twin, MES, ERP, maintenance), infrastructure operations (rail, water, energy). Same spec, different connectors. If you are in one of these markets and want this layer for your stack, the standard is open today — tell us what you need on the hub.
